103  *
104  * Below is an excerpt from an email from Guy Harris on the tcpdump-workers
105  * mail list when someone asked, "How do I Get the length of the TCP
106  * payload?" Guy Harris' slightly snipped response (edited by him to
107  * speak of the IPv4 header length and TCP data offset without referring
108  * to bitfield structure members) is reproduced below:
109  *
110  * The Ethernet size is always 14 bytes.
111  *
112  * <snip>...</snip>
113  *
114  * In fact, you *MUST* assume the Ethernet header is 14 bytes, *and*, if
115  * you're using structures, you must use structures where the members
116  * always have the same size on all platforms, because the sizes of the
117  * fields in Ethernet - and IP, and TCP, and... - headers are defined by
118  * the protocol specification, not by the way a particular platform's C
119  * compiler works.)
120  *
121  * The IP header size, in bytes, is the value of the IP header length,
122  * as extracted from the "ip_vhl" field of "struct sniff_ip" with
123  * the "IP_HL()" macro, times 4 ("times 4" because it's in units of
124  * 4-byte words). If that value is less than 20 - i.e., if the value
125  * extracted with "IP_HL()" is less than 5 - you have a malformed
126  * IP datagram.
127  *
128  * The TCP header size, in bytes, is the value of the TCP data offset,
129  * as extracted from the "th_offx2" field of "struct sniff_tcp" with
130  * the "TH_OFF()" macro, times 4 (for the same reason - 4-byte words).
131  * If that value is less than 20 - i.e., if the value extracted with
132  * "TH_OFF()" is less than 5 - you have a malformed TCP segment.
133  *
134  * So, to find the IP header in an Ethernet packet, look 14 bytes after
135  * the beginning of the packet data. To find the TCP header, look
136  * "IP_HL(ip)*4" bytes after the beginning of the IP header. To find the
137  * TCP payload, look "TH_OFF(tcp)*4" bytes after the beginning of the TCP
138  * header.
139  *
140  * To find out how much payload there is:
141  *
142  * Take the IP *total* length field - "ip_len" in "struct sniff_ip"
143  * - and, first, check whether it's less than "IP_HL(ip)*4" (after
144  * you've checked whether "IP_HL(ip)" is >= 5). If it is, you have
145  * a malformed IP datagram.
146  *
147  * Otherwise, subtract "IP_HL(ip)*4" from it; that gives you the length
148  * of the TCP segment, including the TCP header. If that's less than
149  * "TH_OFF(tcp)*4" (after you've checked whether "TH_OFF(tcp)" is >= 5),
150  * you have a malformed TCP segment.
151  *
152  * Otherwise, subtract "TH_OFF(tcp)*4" from it; that gives you the
153  * length of the TCP payload.
154  *
155  * Note that you also need to make sure that you don't go past the end
156  * of the captured data in the packet - you might, for example, have a
157  * 15-byte Ethernet packet that claims to contain an IP datagram, but if
158  * it's 15 bytes, it has only one byte of Ethernet payload, which is too
159  * small for an IP header. The length of the captured data is given in
160  * the "caplen" field in the "struct pcap_pkthdr"; it might be less than
161  * the length of the packet, if you're capturing with a snapshot length
162  * other than a value >= the maximum packet size.
163  * <end of response>
164  *
